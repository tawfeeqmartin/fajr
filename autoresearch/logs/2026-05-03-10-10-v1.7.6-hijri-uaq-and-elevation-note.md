## Hypothesis

`fajr.hijri()` defaults to the Kuwaiti arithmetic (tabular) calendar which diverges
from the Umm al-Qura calendar used by AlAdhan, IslamicFinder, IACAD, and Microsoft
by 0–1 days routinely and by a full month at Eid boundaries (~38% mismatch rate
in agiftoftime-agent's 16-date audit, issue #48).

Switching the default to Umm al-Qura (via an embedded 31 KB tabular JSON for
1318–1500 AH / 1900–2077 Gregorian) will fix all 16 known mismatches and make
fajr's hijri output match every major Islamic app's reference calendar.

Additionally, the city-registry elevation auto-resolution note (added in v1.7.0
phase 2) lacked magnitude context — users seeing it didn't know HOW MUCH Maghrib
was shifting. The Maghrib note now includes the computed dip magnitude in minutes
(resolves #50 issue from agiftoftime-agent).

## Change made

### Part A — hijri Umm al-Qura (resolves #48)

**New file: `src/data/umm-al-qura-tabular.json`** (31 KB, moved from `scripts/data/`)
Pre-computed two indices from the umalqura/umalqura MIT package (cross-validated
against .NET BCL / ummulqura.org.sa):
- `monthLengths`: per-year 12-element arrays of month lengths (29 or 30 days)
- `startGregorian`: Gregorian start date of 1 Muharram per year (1318–1501)

**New file: `src/hijri-umm-al-qura.js`**
`gregorianToHijriUAQ(date)` — O(log 184 + O(12)) binary-search lookup:
1. Normalize to UTC midnight
2. Range-check: throws RangeError if outside 1900-04-30 / 2077-11-16
3. Binary-search `startGregorian` for largest year whose start ≤ date
4. Accumulate elapsed days, walk `monthLengths` to resolve month + day

**Modified: `src/hijri.js`**
`hijri(date, opts = {})` now dispatches on `opts.convention`:
- `'umm-al-qura'` (default) → `gregorianToHijriUAQ`
- `'tabular'` → original Kuwaiti arithmetic algorithm (preserved for backwards-compat)
- `'observational'` → throws NotImplementedError (planned v1.9.x)

**Modified: `src/index.d.ts`**
Added `HijriConvention` string-union type and `HijriOptions` interface.
Updated `hijri()` signature to `hijri(date: Date, opts?: HijriOptions): HijriResult`.

### Part B — Maghrib elevation note magnitude (resolves #50)

**Modified: `src/engine.js`** (line ~2444)
The city-registry elevation note now reads:
`"Elevation auto-resolved from city registry: {city}, {m}m → Maghrib +X.X min later,
Shuruq -X.X min earlier vs sea-level. Saudi/Umm al-Qura institutionally declines
this correction; UAE (Burj Khalifa) + Malaysia JAKIM apply it. To match Saudi
convention, pass elevation: 0."`

Magnitude computed via existing `computeElevationDipMinutes(elevation, latitude)`.

**Justification:** Umm al-Qura is the official calendar of the Kingdom of Saudi
Arabia (custodian of the two holy mosques) with decades of government publication.
Used by AlAdhan, IslamicFinder, IACAD, Microsoft Windows, Apple (iOS), and Linux glibc
as the de facto global reference. Not pure 🟢 because Diyanet (Turkey), JAKIM
(Malaysia), and regional moonsighting committees legitimately diverge by ±1 day —
Umm al-Qura is the digital ecosystem consensus, not the scholarly consensus of the
full ummah. Disclosed in the docstring and README section.

The old Kuwaiti tabular is also 🟢 in isolation (standard arithmetic calendar),
but its continued use as fajr's default was indefensible given the ecosystem divergence.
Switching to Umm al-Qura is an accuracy improvement, not a scholarly departure.
