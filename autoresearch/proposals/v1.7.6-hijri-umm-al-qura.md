# v1.7.6 — `fajr.hijri()` Umm al-Qura calendar (resolves #48)

## TL;DR

`fajr.hijri()` currently ships the **Kuwaiti tabular** Hijri calendar (30-year
arithmetic cycle, 11 leap years). AlAdhan, IslamicFinder, IACAD, Habibur, and
Microsoft Windows all use **Umm al-Qura** (Saudi Arabia's official
sighting-aligned tables). They diverge by 0–1 days routinely and by a full
month around Eid boundaries.

agiftoftime-agent's audit caught 6/16 mismatches against AlAdhan's
`/v1/gToH`. My follow-up audit caught **25/36 (69.4%)**, including all three
recent Eid al-Fitr month-jumps and both Eid al-Adha day-offs.

This proposal adds an embedded Umm al-Qura table (1318–1500 AH ≈
1900-04-30 → 2077-11-16, all 178 years), changes the default convention,
and keeps the old Kuwaiti tabular reachable via an opt-in flag for backwards
compatibility.

The raw rawData block in the source is a flat array of 4-tuples
`(monthLengthFlags, gy, gm, gd)` per Hijri year. `monthLengthFlags` is a
12-bit bitmask: bit *i* set ⇒ month *(i+1)* of that Hijri year has 30 days,
clear ⇒ 29 days. `(gy, gm, gd)` is the Gregorian date of 1 Muharram for that
Hijri year. The final row in rawData is a sentinel with `flags = 0` whose
date marks the upper bound of coverage.

The pre-staged JSON at `scripts/data/umm-al-qura-tabular.json` (31 KB)
unpacks this into two indices:

- `monthLengths`: `{ "1318": [29|30, ...×12], ... }` — the human-readable shape
- `startGregorian`: `{ "1318": "1900-04-30", ..., "1501": "2077-11-17" }` —
  needed for fast Gregorian→Hijri lookup (binary-search the start dates)

Both are derivable from the bitmask alone, but pre-computing both makes the
runtime lookup branch-free and avoids re-deriving on every call.

## Algorithm

### Gregorian → Hijri (the load-bearing direction)

```
1. Reject if date < 1900-04-30 or date >= 2077-11-17 (out of table coverage)
2. Binary-search startGregorian for the largest hy whose
   startGregorian[hy] <= input date  → call this entry hy0
3. nDays = floor((input - startGregorian[hy0]) / 86400_000)
4. Walk monthLengths[hy0] from m=1 forward, subtracting each month's length
   from nDays until nDays < currentMonthLength
5. Return { year: hy0, month: m, day: nDays + 1 }
```

This mirrors the algorithm `umalqura/umalqura` ships
(`UmAlQuraStatic.gregorianToHijri`), which I verified inline. Lookup is
**O(log 183) + O(12)** = constant-time in practice.

### Hijri → Gregorian (the reverse direction — needed for some app callers)

```
1. Reject if hy < 1318 or hy > 1500
2. dt = startGregorian[hy], remaining = day - 1
3. For m = 1 .. requestedMonth-1: dt += monthLengths[hy][m] days
4. dt += (day - 1) days
5. Return dt
```

Both directions live entirely in JS — no external dep, no async.

## API change

### Default behavior change

```js
// v1.7.5 and earlier (Kuwaiti tabular):
fajr.hijri(new Date('2025-03-30'))
// → { year: 1446, month: 9,  day: 30, monthName: 'Ramadan' }

// v1.7.6 default (Umm al-Qura):
fajr.hijri(new Date('2025-03-30'))
// → { year: 1446, month: 10, day: 1,  monthName: 'Shawwal' }
```

This is a **breaking change in observable behavior** but **not** in API
signature. ~70% of dates we tested return a different value.

### New `convention` parameter

```js
fajr.hijri(date, { convention: 'umm-al-qura' })  // = default = matches AlAdhan
fajr.hijri(date, { convention: 'tabular' })      // = old Kuwaiti tabular (preserved for backwards-compat)
fajr.hijri(date, { convention: 'observational' }) // = future v1.9.x = real moonsight per Yallop/Odeh/Shaukat
```

`'observational'` is **not implemented in v1.7.6** — calling it should throw
a clear `NotImplementedError` mentioning that hilal-based dating already
exists in `fajr.hilal()` and that a full observational Hijri tracker is
planned for v1.9.x.

### Out-of-range handling

The Umm al-Qura table covers 1900-04-30 to 2077-11-16. For dates outside
this window, options:

- **Option A (recommended):** throw `RangeError` with a clear message
  indicating the supported range. Loud failure beats silent wrong answers.
- Option B: silently fall back to Kuwaiti tabular for out-of-range dates.
  Rejected — silent calendar switching is exactly the bug we're fixing.

I recommend **Option A**. Almost no real fajr consumer queries dates
outside 1900–2077 (life expectancy + reasonable forward planning covers it).
For callers who explicitly want pre-1900 / post-2077 dates, they can pass
`{ convention: 'tabular' }` and accept the precision loss.

### Returned shape

Unchanged: `{ year, month, day, monthName }`. Month names stay in the
existing transliteration ("Shawwal", not "Shawwāl") for ASCII-friendliness
unless we're explicitly asked to switch — that's a separate doc-level
question, not load-bearing for this fix.

## Backwards-compat impact

| Aspect | Before | After |
|--------|--------|-------|
| API signature | `hijri(date)` | `hijri(date, opts?)` — opts optional |
| Default output | Kuwaiti tabular | Umm al-Qura |
| Observable diff rate | n/a | ~70% of dates differ from old default by 0–1 days; ~3% of dates differ by a full month (Eid boundaries) |
| Backwards-compat path | n/a | `hijri(date, { convention: 'tabular' })` returns the v1.7.5-and-earlier output exactly |

**Why a patch (1.7.6) and not minor (1.8.0)?** The judgment call:

- The old behavior was demonstrably **incorrect** when measured against the
  rest of the ecosystem (AlAdhan, IslamicFinder, IACAD, Habibur, Microsoft).
  It diverged from public consensus, including from Saudi Arabia's own
  official calendar.
- The new behavior matches what every consumer app expects, which is
  precisely the bug agiftoftime-agent caught in #48.
- Anyone who *specifically* wants Kuwaiti tabular still has a one-line path
  to it: `{ convention: 'tabular' }`.

A patch bump is defensible under "fixing a clearly-incorrect default."
A minor bump is more conservative semver. I lean **patch (1.7.6)** because
keeping the wrong default for a whole minor cycle just to be polite to
semver would mean shipping known-wrong Eid dates in v1.7.x indefinitely.

If reviewers prefer minor (v1.8.0), the implementation is identical;
only the version bump differs.

## Scholarly classification

**🟡→🟢** — Approaching established.

**Justification:**

- **Umm al-Qura** is the official calendar of the Kingdom of Saudi Arabia —
  the custodian state of the two holy mosques. Its rulings on calendar
  matters (including Ramadan/Eid declarations) are followed by a large
  fraction of the global Muslim population, especially through the
  Mecca-Madina link.
- It has decades of institutional use, government publication, and
  cross-validation by computer science (Microsoft's BCL, Apple's calendar,
  Linux glibc, all standardised on it for "Islamic - Umm al-Qura").
- However, "Umm al-Qura" itself is one *of several* legitimate Hijri
  calendars — Diyanet (Turkey), JAKIM (Malaysia), and the various national
  moonsighting committees produce their own schedules that diverge from
  Umm al-Qura by 0–1 days. So Umm al-Qura is not consensus across the
  *ummah*, only consensus within the digital ecosystem.
- The current Kuwaiti tabular default is the *worse* legitimate-disagreement
  choice: it is a purely arithmetic calendar with no institutional
  observance behind it, and it diverges from every major consumer-app
  Hijri calendar.
- The right framing per `feedback_surface_disagreement`: **default to the
  most widely-used, but expose the others as opt-ins**. Umm al-Qura is the
  defensible default; `tabular` and (future) `observational` and (further
  future) `diyanet` / `jakim` should be reachable via the convention flag.

This is not 🔴 (novel) — none of this work introduces astronomy that wasn't
already public and validated. It is not yet pure 🟢 because we are taking a
position on default by following ecosystem-majority rather than scholarly
consensus, and we should disclose that in the README.

## Caveats

1. **Year coverage 1900–2077.** Anyone querying Hijri dates outside this
   window will get a `RangeError`. This is intentional and matches what
   `umalqura/umalqura`, Microsoft, and Apple all do. The Umm al-Qura
   calendar itself only publishes through 1500 AH for now.

2. **Transliteration normalization.** AlAdhan returns "Shawwāl" (with
   macron); we return "Shawwal" (ASCII). For matching/diffing purposes, the
   day + month-number + year are the load-bearing signals; the English
   string is a label only.

3. **Day boundary at midnight UTC vs. local.** Our lookup uses UTC midnight
   for date keys, matching AlAdhan's behavior. If a downstream caller passes
   a `Date` whose timezone interpretation puts it on the *previous* UTC
   civil day, they'll get the previous day's Hijri. This is true in
   v1.7.5 too — no change. Worth a doc note.

4. **The JSON data file adds 31 KB to the bundle.** For a 100 KB-ish
   library this is non-trivial but acceptable; nothing else in fajr is
   embedded data of this size. If bundle size becomes a concern in v2, the
   bitmask format (1 short per year × 183 years = 366 bytes raw) is much
   more compact, but the readability cost is real.

5. **Other Hijri conventions.** Diyanet (Turkey) and JAKIM (Malaysia)
   produce calendars that occasionally diverge from Umm al-Qura by ±1 day.
   The `convention` parameter design leaves space for `'diyanet'` and
   `'jakim'` as future opt-ins without further API churn.
